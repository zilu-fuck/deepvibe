# DeepVibe Core 详细设计文档 v2.1

## 摘要

DeepVibe Core 是一个与界面无关的 AI 编程引擎，专为 DeepSeek 模型深度优化。它以命令行形式交付 MVP，核心目标是让开发者用自然语言在一个已有项目中完成可信赖的代码级修改。本文档整合了 DeepSeek 模型的最新能力（思考模式、Json Output、前缀续写、工具调用、KV Cache 等），对架构与功能进行了全面升级，可直接指导工程落地。

---

## 1. 愿景与初始目标

### 1.1 愿景

打造一个高度智能、安全、可嵌入多种开发环境的 AI 编程引擎核心。未来，无论是 CLI、桌面 App、VS Code 插件还是 MCP Server，都通过同一核心提供一致的 AI 编码体验。

### 1.2 初始目标 (MVP)

> 在一个已有的 Git 项目中，用户提供一句自然语言指令。DeepVibe Core 自动扫描并选取相关文件，调用 DeepSeek 模型生成结构化修改方案，安全地将变更应用到文件系统，并为该次 AI 变更生成可追踪、可撤销的记录。

### 1.3 成功标准

- 正确解析自然语言指令，选出最多 N 个相关文件。
- 调用 DeepSeek API，获得结构化 diff 输出。
- 变更成功写入磁盘，文件内容正确。
- 在干净工作区为 AI 变更生成独立提交；在脏工作区记录 AI 操作补丁，且可安全撤销最近一次 AI 变更。
- 终端输出清晰摘要：修改文件数、增删行数、AI 操作 ID 或提交哈希。

## 4. 核心工作流详解（Engine）

Engine 是总指挥。单次任务的主流程如下：

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  1.初始化 │───▶│  2.扫描  │───▶│ 3.构建上 │───▶│ 4.模型调 │───▶│ 5.应用变 │
│  加载配置 │    │  选择文件 │    │   下文    │    │   用     │    │   更     │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
                                                                       │
                                                              ┌────────┘
                                                              ▼
                                                       ┌──────────┐
                                                       │ 6.输出报 │
                                                       │   告     │
                                                       └──────────┘
```

### 4.1 初始化

- 检查当前目录是否为 Git 仓库（需 `git init` 或拒绝执行）。
- 读取项目配置文件：
  - `~/.deepvibe/config.json`（全局配置：API Key、默认模型）
  - `.deepvibe/config.json`（项目级配置：忽略规则、自定义 Prompt）
- 若未配置 API Key，引导用户设置。

### 4.2 项目扫描

`Scanner` 模块负责生成候选文件列表。

**输入**：
- 项目根目录路径
- 用户指令文本（用于提取 `@file` 引用）
- 配置的忽略规则

**处理逻辑**：
1. 递归遍历目录，排除 `node_modules`、`.git`、`dist`、`build` 等。
2. 应用 `.gitignore` 和 `.deepvibeignore`。
3. 为每个文件打分：
   - 最近 Git 修改的文件：+10
   - 文件名/路径包含指令关键词：+8
   - 被测文件有对应的源文件：+3
   - 按分数降序排序，取前 5 个（可配置数量）。
4. 若指令中有 `@src/utils/api.ts`，该文件强制加入候选列表。
5. 输出候选文件路径数组。

**输出**：`string[]`（相对路径列表）

### 4.3 上下文构建

`Context Builder` 负责组装发送给模型的完整 `messages` 数组，这是整个系统最关键的性能影响者。

#### 4.3.1 KV Cache 优化策略

保持消息前缀稳定，最大化 DeepSeek KV Cache 命中率。消息序列结构如下（固定前缀 + 可变后缀）：

```
┌──────────────────────────────────────────────────┐
│ 消息 1: system                                   │
│ ┌──────────────────────────────────────────────┐ │
│ │ 角色定义 + 输出格式要求 + 工具定义             │ │  ▶ 完全固定
│ └──────────────────────────────────────────────┘ │
├──────────────────────────────────────────────────┤
│ 消息 2: user (项目元信息)                         │
│ ┌──────────────────────────────────────────────┐ │
│ │ 项目语言、框架、文件树摘要、核心约定            │ │  ▶ 同项目内基本稳定
│ └──────────────────────────────────────────────┘ │
├──────────────────────────────────────────────────┤
│ 消息 3: user (当前任务 + 相关文件)                │
│ ┌──────────────────────────────────────────────┐ │
│ │ 任务描述、选中文件内容、联网搜索结果            │ │  ▶ 每次变化
│ └──────────────────────────────────────────────┘ │
├──────────────────────────────────────────────────┤
│ 消息 4..n: 历史对话 (如有)                        │
│ ┌──────────────────────────────────────────────┐ │
│ │ assistant 回复、tool 调用、tool 结果           │ │  ▶ 不断增长
│ └──────────────────────────────────────────────┘ │
├──────────────────────────────────────────────────┤
│ 消息 n+1: user (新指令)                          │  ▶ 当前轮
└──────────────────────────────────────────────────┘
```

**效果**：系统 Prompt（消息 1）和项目元信息（消息 2）在不同轮次间几乎不变。重写这部分来获得更好的缓存命中率。

#### 4.3.2 Token 预算管理

```
官方模型能力上限:
├─ Context Length: 1M tokens
└─ Max Output: 384K tokens

M1 保守软预算（默认实现）：
├─ 固定前缀（系统 Prompt + 项目元信息） : ~3K tokens
├─ 预留思考与响应空间                  : ~32K tokens
└─ 文件内容与对话历史可用空间           : ~165K tokens
```

**裁剪策略**（当超限时）：
1. 移除文件中的注释和空行。
2. 裁剪函数体，只保留签名。
3. 减少附带文件数量。
4. 压缩历史对话，仅保留最近 3 轮。

#### 4.3.3 系统 Prompt 模板

```typescript
const SYSTEM_PROMPT = `
你是一位经验丰富的全栈软件工程师，专注于根据需求对项目代码进行精确修改。

## 核心原则
- 只修改与需求直接相关的代码，不要重构无关部分。
- 保持现有代码风格和架构约定。
- 修改后必须保证类型安全和导入完整。
- 不要输出任何解释，只输出符合格式要求的 JSON。

## 输出格式
你必须在 content 中输出合法的 JSON 对象，格式如下：
{
  "files": [
    {
      "path": "相对路径",
      "action": "modify | create | delete",
      "diff": "unified diff 内容，包含完整上下文"
    }
  ],
  "summary": "简要描述修改了什么"
}
`;
```

### 4.4 模型调用

`DeepSeekClient` 封装 API 通信。

**关键参数**：

| 参数 | 设置 | 说明 |
|------|------|------|
| `model` | `deepseek-v4-pro` | 默认高质量模型，支持 thinking on/off |
| `messages` | Builder 构建的数组 | |
| `stream` | `true` | 启用流式输出 |
| `response_format` | `{"type": "json_object"}` | 强制输出合法 JSON |
| `thinking` | `{"type": "enabled"}` | 开启思考模式 |
| `reasoning_effort` | `"high" \| "max"` | 控制推理强度；默认 `high`，`--deep` 使用 `max` |

**M1 决策**：
- 默认链路使用标准 Chat Completions + `response_format: { type: "json_object" }`。
- Chat Prefix Completion 不纳入 M1 默认链路；仅保留为后续实验能力。
- 若后续启用 Prefix Completion，必须满足两个条件：
  - `messages` 最后一条消息为 `assistant`，且该消息设置 `prefix: true`
  - `base_url` 切换到 `https://api.deepseek.com/beta`

**流式处理**：
- `reasoning_content` delta：输出到终端为浅灰色，展示模型思考过程。
- `content` delta：累积到缓冲区，最后整体交解析器。

**错误处理**：
- 网络重试 3 次（指数退避）。
- 429 限速时等待 `Retry-After`。
- 超时 90 秒。

### 4.5 响应解析

`ResponseParser` 将模型输出转为可操作的修改列表。

1. 收集完整的 `content`、`reasoning_content`、`finish_reason` 与 `usage`。
2. 若 `content` 为空，则标记为可恢复错误，缩小上下文后重新请求一次。
3. 若 `finish_reason = "length"`，则视为输出截断；优先提高 `max_tokens` 或缩小上下文，并重新请求一次。
4. 对 `content` 执行 `JSON.parse`。
5. 校验 JSON 结构是否符合 Schema；如失败，发起一次带错误说明的修复性请求。
6. 提取 `files` 数组，验证每个 `path` 是否在项目目录内。
7. 若恢复后仍失败，则终止流程，不进入文件写入阶段。

### 4.6 变更应用

`Patcher` 负责将 diff 写入文件系统。

**流程**：
1. 计算项目根目录的规范化真实路径（realpath）。
2. 对每个目标文件，基于 `path.resolve` + `fs.realpath`（或最近存在祖先目录的 realpath）验证其真实落点仍在项目根目录内。
3. 拒绝绝对路径、`..\\` / `../` 逃逸、UNC 路径，以及通过 symlink / junction 指向仓库外部的目标。
4. 对每个待修改文件，读取当前内容作为备份（内存中）。
5. 使用 `diff` 库的 `applyPatch` 应用 unified diff。
6. 若任何文件应用失败，回滚已修改的所有文件，报告错误。
7. 全部成功后，交由 Git Manager 根据工作区状态创建 AI 提交或写入 AI 操作记录。

## 5. 关键策略与决策

### 5.1 思考模式上下文管理策略

| 场景 | 处理方式 | 原因 |
|------|----------|------|
| 普通对话轮次 | 不传入上一轮 `reasoning_content` | 官方建议，避免缓存失效 |
| 同层工具调用链 | 必须传入产生该调用的 `reasoning_content` | API 要求，保证链式调用的完整性 |
| 历史对话压缩 | 清洗已完成的推理内容 | 释放窗口空间 |

### 5.2 模型选择策略

| 任务特征 | 使用模型 | 思考模式 |
|----------|----------|----------|
| 默认 | `deepseek-v4-pro` | `thinking: enabled` + `reasoning_effort: high` |
| 用户指定 `--flash` | `deepseek-v4-flash` | `thinking: enabled` + `reasoning_effort: high` |
| 用户指定 `--deep` | `deepseek-v4-pro` | `thinking: enabled` + `reasoning_effort: max` |

### 5.3 安全措施

| 层次 | 措施 |
|------|------|
| 路径 | 使用 `resolve + realpath` 校验真实落点必须位于项目根目录内；拒绝 UNC、symlink、junction 与目录逃逸 |
| 内容 | 拒绝修改 `.git/` 和 `.deepvibe/` 下的任何文件 |
| 操作 | 绝不使用 `git reset --hard` 作为 `undo` 机制；仅撤销最近一次 AI 成功变更 |
| 确认 | 默认展示 diff 并要求确认，`--force` 可跳过 |

### 5.4 Git 与 Undo 策略

为避免误伤用户已有工作区改动，M1 将 `undo` 语义定义为“撤销最近一次成功的 AI 变更”，而不是“回退整个仓库到某个旧提交”。

**干净工作区**：
- AI 变更应用成功后，创建一条仅包含 AI 变更的独立提交。
- `deepvibe undo` 通过 `git revert --no-edit <ai_commit>` 撤销该提交。

**脏工作区**：
- 不自动提交用户已有未提交改动。
- AI 变更应用成功后，在 `.deepvibe/operations/<operation-id>.json` 中记录：
  - 操作 ID
  - 基准 `HEAD`
  - 触达文件列表
  - 每个文件的 `pre-image hash`
  - 每个文件的 `post-image hash`
  - forward diff / reverse diff
- `deepvibe undo` 仅对最近一次成功操作生效，并且只有在目标文件仍匹配记录中的 `post-image hash` 时才执行反向补丁；若文件已被再次修改，则拒绝自动回滚并提示人工处理。

## 8. 附录

### A. 与 DeepSeek API 集成的关键细节

- 思考模式中，`reasoning_content` 在普通多轮对话里无需保留；但在同级工具调用链中必须回传，否则 API 会返回 400。
- `response_format: { type: "json_object" }` 只能保证 JSON 输出能力可用；仍需在 prompt 中明确要求 JSON，并提供示例格式。
- JSON Output 存在偶发空 `content` 风险，M1 解析链路必须显式处理。
- KV Cache 对所有用户默认开启，无需额外配置，只需保持消息前缀稳定即可受益。
- Chat Prefix Completion 需要 `assistant` 前缀消息、`prefix: true`，并切换到 `https://api.deepseek.com/beta`；暂不纳入 M1 默认链路。
- FIM 补全需调用独立的 `/beta/completions` 端点，与 Chat API 不同，暂不纳入 M1。

## 核心模块实现要点

### Scanner（项目扫描）
- 使用 `fast-glob` 遍历文件，默认忽略 `node_modules`、`.git`、`dist`、`build`。
- 读取 `.gitignore` 和项目级的 `.deepvibeignore`。
- 用 `simple-git` 获取最近修改的文件，加权排序。
- 解析用户指令中的 `@路径` 引用，强制包含。

### Context Builder（上下文构建）
- 固定前缀：系统 Prompt（角色定义 + JSON 输出格式要求）→ 项目元信息（语言、框架、文件树摘要）。
- 可变后缀：选中文件内容 + 当前用户指令。
- Token 预算：tiktoken 精确估算，超限时依次裁剪注释、函数体、文件数量。
- KV Cache 友好：系统 Prompt 和项目元信息在整个会话中保持完全不变。

### DeepSeek Client（模型调用）
- 使用 `fetch` 直接调用 DeepSeek Chat Completions API。
- 请求参数：`model: "deepseek-v4-pro"`，`stream: true`，`response_format: { type: "json_object" }`。
- 在请求体顶层传入 `thinking: { type: "enabled" }`，并显式设置 `reasoning_effort`。
- 内置重试机制：最多 3 次重试，指数退避，支持 `Retry-After` 响应头，429 和 5xx 状态码自动重试。
- M1 默认不依赖 Prefix Completion；若后续引入，需走 beta base URL，并把最后一条 `assistant` 消息设为 `prefix: true`。
- 流式处理：`reasoning_content` 累积到缓冲区，`content` 累积后统一交解析器。

### Response Parser（响应解析）
- 对累积的 content 做 `JSON.parse`。
- 同时校验 `finish_reason`、空 `content`、schema 结构是否合法。
- 若解析失败，按“截断重试 -> 修复性重试 -> 终止”的顺序处理。

### Patcher（变更应用）
- 验证每个目标路径的 realpath 落在项目目录内，并拒绝 symlink / junction 逃逸。
- 使用 `diff` 库的 `applyPatch` 逐个文件应用 unified diff。
- 任一文件失败则内存回滚所有已改文件。
- 全部成功后，调用 Git Manager 创建 AI 提交或记录 AI 操作。

### Git Manager（版本控制）
- 干净工作区：修改后创建仅包含 AI 变更的提交。
- 脏工作区：写入 `.deepvibe/operations/<operation-id>.json`，不自动提交用户原改动。
- Undo：干净工作区使用 `git revert`；脏工作区使用带哈希校验的 reverse diff。

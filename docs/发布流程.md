# DeepVibe Core 发布流程

适用范围：

- 发布 npm CLI 包
- 本地验证可发布产物
- 本地安装 smoke test
- 启动服务模式后的基础部署检查
- 项目内插件交付时的最小检查

## 1. 发布前检查

在仓库根目录执行：

```bash
pnpm install
pnpm release:check
pnpm release:smoke
```

这一步会依次执行：

- `pnpm check`
- `pnpm test`
- `pnpm build`
- `pnpm pack:dryrun`
- `pnpm release:smoke` 会额外执行一次“打包产物安装 -> 命令可执行”验证

目标：

- 类型检查通过
- 自动化测试通过
- `dist/` 可正常构建
- npm 包内容符合预期
- 打包产物安装后，`deepvibe` 命令可执行

## 2. 检查将被打包的文件

先看 `package.json` 的 `files` 和 `bin`：

- `files`
- `bin.deepvibe`

再执行：

```bash
pnpm pack:dryrun
```

重点确认：

- `dist/index.js` 在包里
- `README.md` 在包里
- 没有把测试、临时文件、工作目录垃圾打进包

## 3. 版本发布步骤

建议顺序：

1. 更新版本号
2. 跑完整发布前检查与本地安装验证
3. 打 tag
4. 发布 npm

示例：

```bash
# 例：先手动改 package.json version
pnpm release:check
pnpm release:smoke
git add -A
git commit -m "Release: v0.1.0"
git tag v0.1.0
npm publish
```

如果使用受保护分支，先通过 PR 合并，再在发布分支打 tag。

## 4. 本地安装验证

推荐直接运行：

```bash
pnpm release:smoke
```

它会：

- 执行 `npm pack`
- 把 tarball 安装到临时目录
- 验证 `deepvibe --help`
- 验证 `deepvibe serve --help`
- 清理临时目录和生成的 tarball

如果你想手工确认，也可以这样做：

```bash
npm install -g deepvibe-core
deepvibe --help
```

如果你不想污染全局环境，也可以直接用打包产物测试：

```bash
npm pack
npm install -g ./deepvibe-core-0.1.0.tgz
deepvibe --help
```

## 5. 发布后检查

发布完成后，至少再次确认：

- 从实际安装环境中执行 `deepvibe --help`
- 如本次发布涉及服务相关改动，再执行一轮服务模式最小检查

## 6. 服务模式部署检查

服务模式当前是内建 Node HTTP 服务。

最小启动：

```bash
deepvibe serve --host 127.0.0.1 --port 4242
```

最小健康检查：

```bash
curl http://127.0.0.1:4242/health
```

最小 JSON-RPC 检查：

```bash
curl -X POST http://127.0.0.1:4242/rpc ^
  -H "Content-Type: application/json" ^
  -d "{\"jsonrpc\":\"2.0\",\"id\":1,\"method\":\"deepvibe.health\"}"
```

如果部署到远程环境，至少要额外确认：

- 监听地址是否正确
- 反向代理是否允许 SSE
- 超时配置不会截断 `/tasks/:taskId/events`

## 7. 插件交付检查

项目内插件最小结构：

```text
.deepvibe/
  plugins/
    my-plugin/
      plugin.json
      index.js
```

发布或共享插件前至少确认：

- `plugin.json` 可解析
- `entry` 路径存在
- 工具定义名称唯一
- 权限字段最小化
- `runtime.timeoutMs` 和 `runtime.memoryLimitMb` 合理

如果插件用于服务模式，还要确认：

- `permissions.allowInService` 是否真的需要打开

## 8. 已知发布限制

当前发布层还不完整，主要限制：

- 没有自动 changelog 生成
- 没有自动版本策略脚本
- 没有服务部署模板
- 没有插件市场或插件版本兼容矩阵

## 9. 推荐的最小发布清单

每次发布前，至少逐项确认：

- 版本号已更新
- `pnpm release:check` 通过
- `pnpm release:smoke` 通过
- README 已反映当前功能状态
- 文档里的“当前未完成”已更新
- 包内容检查无误
- 标签和发布说明已准备

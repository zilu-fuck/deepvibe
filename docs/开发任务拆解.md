## P0：设计阻塞项

### T0-1 回滚语义重设计

- 优先级：`P0`
- 目标：把“撤销本次 AI 变更”与“回退整个仓库历史”彻底分开。
- 背景：当前方案使用 `git add -A && git commit` 和 `git reset --hard HEAD~1`，会误伤用户已有未提交改动。
- 交付物：
  - 一份明确的回滚语义说明。
  - `dirty working tree` 下的处理策略。
  - `undo` 的实现方案草案。
- 建议方案：
  - 默认不自动提交用户已有未提交改动。
  - 将 AI 变更记录为“本次操作生成的补丁集”或“专用临时引用”，而不是直接假设可以 `reset --hard`。
  - `undo` 只撤销最近一次 AI 成功应用的变更。
- 验证：
  - 在已有本地改动的仓库里执行一次 AI 修改，再执行 `undo`，用户原改动保持不变。
- 预估：`0.5-1 天`
- 依赖：无

### T0-2 DeepSeek API 接入契约校正

- 优先级：`P0`
- 目标：把设计里的模型能力、Prefix Completion、JSON 输出和 thinking 用法校正到当前官方接口。
- 背景：当前计划中 `prefix` 的用法和 `--flash` 模型映射都不准确。
- 交付物：
  - 一份 API 参数表。
  - `pro / flash` 与 `thinking on/off` 的正式映射规则。
  - 是否使用 beta base URL 的明确决策。
- 必须明确：
  - `--flash` 是否映射到 `deepseek-v4-flash`。
  - Prefix Completion 是否纳入 M1。
  - JSON 输出是否只依赖 `response_format`，还是配合额外恢复策略。
- 验证：
  - 用一份最小请求样例跑通真实 API。
- 预估：`0.5 天`
- 依赖：无

### T0-3 路径与文件写入安全模型设计

- 优先级：`P0`
- 目标：补齐 Windows、符号链接、junction、UNC 路径等场景下的真实写入边界。
- 背景：当前只拦截绝对路径和 `../`，不足以防目录逃逸。
- 交付物：
  - 路径解析与校验规则。
  - `realpath`/规范化后再判断根目录包含关系的设计说明。
  - 禁止写入的系统目录清单。
- 验证：
  - 针对 `..\`、绝对路径、符号链接、junction 的测试用例都被拒绝。
- 预估：`0.5 天`
- 依赖：无

### T0-4 响应恢复策略补充

- 优先级：`P0`
- 目标：把“模型偶发输出不稳定”从致命失败降级为可恢复异常。
- 背景：当前只写了 `JSON.parse` 失败后重试一次，强度不够。
- 交付物：
  - 基于 `finish_reason`、空 `content`、截断、schema 不匹配的恢复策略。
  - 是否支持“修复性二次请求”的规则。
- 建议至少覆盖：
  - `content` 为空。
  - 输出被截断。
  - JSON 合法但 schema 非法。
  - `diff` 缺失或 action 非法。
- 验证：
  - 4 类异常输入都能进入确定性的错误分支或恢复分支。
- 预估：`0.5 天`
- 依赖：T0-2

## M1 完成定义

满足以下条件，才算 `M1 done`：

- 可以在一个 Git 仓库中运行 `deepvibe "..."`。
- 工具不会误删用户已有未提交改动。
- 输出解析失败不会进入写盘阶段。
- 路径逃逸、符号链接逃逸、系统目录写入被拒绝。
- 用户能看到清晰的变更摘要，并可撤销最近一次 AI 变更。
- 最低测试矩阵全部通过。
